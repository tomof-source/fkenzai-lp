## 1. 目的・背景

### 1-1. なぜこの再設計が必要か
- 現状の style-shade は「記事型LP」として作ったはずが、結果として 17セクションのフル装備な縦長LP（セールスレター型）になってしまった
- 岡村さん指摘：「記事LPから、どんどん普通のLPに寄っていっていないか」
- AI分析の結論：今のページはヒーローで結論（製品名・LINE CTA）を出し、機能・事例・スタッフ・選ばれる理由・費用・流れ・FAQ …とLPの定番ブロックがフルセットで並んでいる。これは「もう興味がある人を申込みまで運ぶ」公式LP型の構成
- 福嶋判断：参考にした成功事例（記事型）の核は「広告から飛んできた人が、売り込みと気づかずに読み進めてしまう」点。これを取り戻したい

### 1-2. 解決したい課題
- 流入想定が「冷たい見込み客」（リスティング／SNS広告着地）なのに、構成が「温かい見込み客」向けLPになっている
- 同一メッセージ「窓の外で止めるのが効く」が4回繰り返されている（pain-scene／快適／原因の説明／代表挨拶）
- 3〜5万円・1時間施工という低単価商材に対して、スタッフ8人の顔写真は情報量が重すぎる
- 「記事として読み進める導線」と「LP的に申込みを煽る導線」が同居していて、どっちつかず

### 1-3. ゴール
- 冷たい見込み客が「広告→記事→気づいたら商品の話→LINE追加」と自然に進む構成にする
- style-shade で v3 を完成させ、これを横展開の正本テンプレにする（既存の plast-boon／mado-reno-2026 も将来 v3 化を検討）
- TAS ART が仕上げ・f-kenzai.com へのアップを行えるレベルまで叩き台として整える

## 3. 機能要件

### 3-1. 流入想定とCV
- 主要流入：リスティング広告／SNS広告からの冷たい見込み客
- 唯一のCV：LINE友だち追加（`https://lin.ee/UmuaYxJ`）
- 電話CV は副次扱い（フッター近辺のみ）

### 3-2. 記事型としての設計原則
記事型の核は「読み物として始まり、自然に商品の話に着地する」こと。これを構造で担保する：

- 冒頭で製品名・価格・LINE CTA を強く出さない（広告着地で「あ、広告だな」と気づかれない）
- 最初の数スクロールは「読者の悩みに寄り添う読み物」として成立させる
- 商品名（スタイルシェード）の初出は中盤以降（記事の3分の1〜半分を過ぎたあたり）
- LINE CTA は中盤以降に初登場し、最終CTAで強く押す
- 「公式LPの定番ブロック（マイスター紹介／選ばれる理由／会社情報など）」は記事の後半・脚注的な扱いに格下げ

### 3-3. v3 セクション骨格（記事型）
上から下へ、読み物→商品→申込みの3層構造で並べる：

#### Aブロック：読み物（冷たい見込み客の関心を引く）
1. HERO（記事タイトル風）
   - 製品名を出さない。読者の体感ワード（例：「夕方、西日で部屋がサウナになる家。なぜ年々ひどくなるのか」）
   - LINE CTA は置かない／小さくスクロール誘導だけ
2. 導入：読者の状況描写（pain-scene 相当を「読者目線のエッセイ」化）
   - 「うちもそうです」と思わせる具体描写
3. 原因の解説（zukai 図解）
   - 「なぜ起きるのか」を技術的に分かりやすく
   - ここで初めて「窓の外で遮るのが効く」という結論に着地する（メッセージ重複の解消）

#### Bブロック：商品紹介（自然な着地）
4. 解決策の紹介（製品名はここで初出）
   - 「窓の外で遮るには何があるか→スタイルシェード」
   - 効果数値（cmp-flow）はここに統合
5. 施工事例（3件）＋お客様の声（実例のみ・3〜5件に絞る）
6. 費用と流れ（cost-card ＋ steps を統合した1セクション）
   - 「思ったより安い・早い」を感じさせる短いセクション

#### Cブロック：申込み判断材料（脚注的に）
7. FAQ（記事型では「読者の最後の不安解消」として残す）
8. 信頼の補強（短く）
   - Googleクチコミ／代表メッセージ／会社情報を1セクションに圧縮
   - マイスター8人グリッドは v3 では削除候補（または「施工担当」として施工事例カード内に1人ずつ表示する形に変更）
   - YouTube セクションは v3 では削除候補（記事内のリンクで代替）
9. 最終CTA（LINE追加）＋ スマホ追従LINEバーは継続

### 3-4. 重複メッセージの整理ルール
- 「窓の外で止めるのが効く」というコアメッセージは、原因解説セクションでの結論として1回だけ提示する
- 代表メッセージは v3 では本文を「窓のプロとして」「地域への思い」など別軸の内容にする（コア結論の繰り返しを避ける）
- pain と 快適 の対比は HERO 直下の導入で1回完結させる

### 3-5. 過剰要素の整理（福嶋判断 9-1〜9-3 確定）
- マイスター8人グリッド → **削除**。C ブロック信頼補強の中に「施工は当社認定マイスター8名が担当」という1行＋人数だけ残す
- YouTube 2枚サムネブロック → **小型化**。FAQ の前に「動画でもっと知りたい方へ」として1枚だけ残す
- Googleクチコミ巨大ヒーロー（4.8の黄色数字）→ **圧縮統合**。C ブロック信頼補強の中に「Googleクチコミ ★4.8（◯件）」と1行で表示＋外部リンクボタン
- 選ばれる理由（数字カード4枚）→ 解決策セクションに統合

### 3-6. CTA 配置ルール
- HERO：CTA なし（スクロール誘導のみ）
- A ブロック（読み物部分）：CTA なし
- B ブロック先頭（解決策初出）：軽めの LINE CTA（「詳しく聞いてみる」程度）
- B ブロック末尾（費用と流れ後）：標準の LINE CTA
- C ブロック末尾（最終CTA）：強い LINE CTA ＋ 電話
- スマホ追従 LINE バー：B ブロック以降のみ表示（A ブロックでは非表示。記事感を壊さない）

## 4. 非機能要件

### 4-1. パフォーマンス
- 画像は WebP もしくは適切に圧縮された JPG／PNG
- HERO 画像は LCP 対策で 200KB 以下を目標
- ファーストビュー表示まで 2 秒以内（4G 想定）

### 4-2. 表示・対応環境
- スマホファースト（広告流入の8割以上想定）
- iOS Safari 最新 / Android Chrome 最新 で崩れないこと
- PC は副次。1080px 幅以上で読みやすければ可

### 4-3. SEO / 計測
- noindex は外す前提（広告着地だが、検索流入も拾える状態にする）
- OGP は記事タイトル風に差し替え
- GA4 ／ Meta Pixel タグ設置箇所をテンプレに用意（TAS ART が値を入れる）

### 4-4. アクセシビリティ
- 画像 alt 必須
- LINE CTA ボタンは aria-label 付与
- コントラスト比 WCAG AA 準拠

## 6. エッジケース・例外処理

### 6-1. 流入別の振り分け（福嶋判断 9-5 確定）
- 本要件 v3 は「冷たい見込み客（広告流入）」前提で設計
- 現行 v2（`style-shade/`）は残置。公式サイト内リンク／指名検索など温かい見込み客向けに使い分け可能
- 広告出稿側でリンク先 URL を v3 に切り替える運用は別途決定

### 6-2. 「記事感が壊れる」リスク
- TAS ART 仕上げ段階で、装飾・CTA・バナーが過剰に入ると記事型が崩れる
- 制作申し送り時に「記事として読めるかどうか」を最終チェック観点として明示する

### 6-3. グランドルール継承
- 本文に井桁（#）とアスタリスク（*）を使わない
- です・ます調、煽り語禁止
- お客様の声は実例のみ（創作禁止）

---

## 7. 技術的な前提・制約

- 既存 v2 テンプレ（`style-shade/index.html`）の CSS コンポーネントは流用可能なものは活用
- 流用する：`.scene` `.scene-bg` `.scene-in` `.cmp-flow` `.case-list` `.case-card` `.cost-card` `.steps` `.sticky-bar` `.yt-video`（1枚版で）
- 削る／変える：`.master-grid` `.master-card` `.google-hero`（圧縮版へ） `.yt-block` `.yt-grid` `.stat-grid`
- 計測タグ（GA4／Meta Pixel）は v3 では実装しない（TAS ART 仕上げで対応・福嶋判断 9-7）
- 単一HTMLファイル構成は維持（外部CSS/JSに分割しない）
- GitHub Pages 配信（main直push禁止・作業ブランチ→PR→手動マージ）
- ローカル：`C:\Users\fkenz\fkenzai-lp\style-shade-v3\`（新規作成）

## 8. 受け入れ条件（Acceptance Criteria）

### 8-1. 構成として記事型に戻っているか
- HERO に製品名「スタイルシェード」が出てこない
- HERO に LINE CTA ボタンが置かれていない
- 製品名の初出が縦スクロール位置で「全体の3分の1〜半分」以降
- LINE CTA の初出が縦スクロール位置で「全体の半分」以降
- セクション数が v2 の 17 から大幅に減っている（目標 9 前後）

### 8-2. 重複が解消されているか
- 「窓の外で止める」というコアメッセージが、記事内で1回だけ結論として提示される
- 代表メッセージが「窓の外で止める」の繰り返しになっていない

### 8-3. 冷たい見込み客向けに最適化されているか
- 最初の3スクロール分が「読み物として成立」している（広告色を出さない）
- スマホ追従LINEバーが Aブロックでは非表示
- 押し売り感のあるコピー（「今すぐ」「限定」など）が入っていない

### 8-4. 横展開できる正本になっているか
- 商品固有部分（HERO見出し／pain訴求／効果数値／対応エリア）の差し替えポイントが index.html 内でコメント明示されている
- TAS ART が読んで仕上げ作業に入れる粒度のコード・コメントになっている

### 8-5. 実装後の確認
- GitHub Pages 反映後、スマホ（iOS Safari／Android Chrome）で実機確認
- 公開URL `https://tomof-source.github.io/fkenzai-lp/style-shade/?v=N` でキャッシュ回避確認
- 岡村さん（コンサル）に「記事型に戻っているか」レビュー依頼
